### <a name="_3bfxc9a45514"></a>**Функциональные требования**

#### Проблематика:
Необходимо разделить монолитное приложение на микросервисную архитектуру
Исходная схема архитектуры находится в схеме [С3_Hotelio_as_is.puml](diagrams/%D0%A13_Hotelio_as_is.puml)

- Явных требований по проблематике нагруженности сервисов не предоставлено.
- Явной бизнес-ценностной оценки выделение определенного сервиса не предоставлено

#### Текущие проблемы
- Сложность сопровождения
- Любое изменение требует понимания всей кодовой базы.
- Невозможно менять один модуль, не затрагивая другие.
- Низкая масштабируемость
- При пиковых нагрузках (например, на бронирование) нельзя масштабировать только нужный компонент.
- Производительность неравномерна.
- Невозможность гибкой разработки
- Разные команды не могут работать независимо.
- Трудно внедрить быстрый CI/CD.
- Ограничения на фронтенде
- REST API монолита слишком универсален и плохо адаптируется под разные платформы.
- Нет поддержки BFF.
- Сложности с запуском новых фич
- Большой риск ошибок из-за плотной связанности.
- Тестирование требует понимания всех зависимостей.

### <a name="_qmphm5d6rvi3"></a>**Решение**

Сначала будет выделен сервис бронирований (BookingService) в отдельный микросервис.
Функционал из монолита не будет удален сразу.
Будет организована маршрутизация запросов в новый микросервис.
Со временем функционал будет вытеснен из монолита.

Почему будет выделен этот сервис:
 - Представляет собой один из более ценных бизнес-элементов решения
 - Один из нагруженных сервисов
 - От него не зависят другие сервисы

Целевая схема выделения сервиса на схеме [С3_Hotelio_BookingService_be_to.puml](diagrams/%D0%A13_Hotelio_BookingService_be_to.puml)
